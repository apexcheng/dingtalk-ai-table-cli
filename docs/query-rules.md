# 查询规则

## 核心约束

- `query_records` 不适合做大表全量扫描，不要依赖 `cursor` 连续翻页。
- 不要把“排序 + 过滤 + 翻页”串成连续遍历，也不要单独依赖“排序 + 翻页”或“过滤 + 翻页”做全量处理。
- 批量处理只走“第一页 -> 处理记录 -> 回写辅助标记字段 -> 下次查未标记数据”的方式。
- `filters` 必须使用 `fieldId`。
- `singleSelect / multipleSelect` 过滤必须传 option `id`，不是 option name。

## 日期过滤

如果用户只需要按日期过滤，优先按日期查。

支持：

```text
2026-05-01 ✅
```

不支持或不可依赖：

```text
2026-05-01 01:00 ~ 2026-05-01 02:00 ❌
```

如果用户要求小时、分钟、秒级时间范围，先按日期查询，再在本地代码里继续筛选。

## 排序

排序只允许用于确定当前第一页的处理优先级，不要依赖排序 + 翻页完成全量遍历。

示例：

```json
{
  "sort": {"operator": "desc", "fieldName": "采集时间"},
  "limit": 100
}
```

处理完当前第一页后，应回写辅助标记字段，下次查询时排除已标记数据。

## 图片 / 附件字段

- 默认只返回非图片 / 非附件字段。
- 查询前先通过字段列表识别 `attachment`、图片、截图、凭证等字段。
- 如果当前 MCP schema 支持指定返回字段，则只传入非图片 / 非附件字段的 `fieldId`。
- 如果当前 MCP schema 不支持指定返回字段，则查询后只使用非图片字段，并在必要时提示图片字段可能导致查询变慢。

## 辅助标记字段

如果表里没有可用的回写字段，Agent 可以自动新增一个辅助字段，例如 `处理标记`、`查询标记`、`同步标记`、`回查标记` 或 `AI处理标记`。

辅助字段只用于避免重复读取和模拟稳定分页，不应承载业务含义。
